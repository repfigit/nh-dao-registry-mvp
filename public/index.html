Plain-language walkthrough

What this app does from screen to blockchain

The registry is a working full-stack demonstration. The form collects public evidence, the server turns it into signed public identity documents, and the verifier checks each piece independently.

1
Browser collects the filing

The page checks obvious mistakes before submission, such as a missing DAO suffix, a PO box address, malformed URLs, or missing attestations.

2
Server validates again

The Express API repeats the rules on the backend. Client-side checks are helpful, but the server is the authority.

3
Evidence becomes signed DID documents

The app builds two linked did:web documents: one for the DAO and one for the registered agent. The registry signs both with its controller key.

4
Files and hashes make the record verifiable

Governance bytes are content-addressed with IPFS. The canonical hash of each DID document can be anchored on Polygon Amoy when chain settings are configured.

Full-stack map
Frontend

Static HTML and JavaScript in public/ render the form and inspector.

API

POST /api/file publishes records. Admin review decisions use protected /api/admin routes.

Storage

The MVP stores records and local IPFS blobs on the Fly volume mounted at /app/data.

Identity

DID documents are served from normal HTTPS URLs so anyone can resolve them from the DID alone.

Verification

The verifier does not trust the UI. It re-resolves documents, checks signatures, links, IPFS hashes, and chain anchors.

Step 1 of 5

DAO identity

What you are doing: naming the DAO and providing the public materials that explain the DAO's rules, code, and user interface.

What the app does next: if you upload bylaws or a governance file, the backend pins those exact bytes for permanent storage. If you only provide a URL, the MVP records the URL but does not download it.

For this MVP: upload the PDF or governance file you want preserved. Optional URLs must be public HTTPS URLs; localhost, private network URLs, and placeholder links will be rejected by the server.

Registered agent

What you are doing: identifying the person or entity that can receive official notices for the DAO.

What the app does next: the backend creates a second DID document for the registered agent and links it bidirectionally to the DAO DID. That lets the verifier confirm that the DAO and agent records point to each other.

Important: the address needs to be a physical New Hampshire street address. A PO box is not enough for the registry workflow.

This person or entity is the official contact for legal service. The address must be a physical New Hampshire street address.

Smart contracts

What you are doing: listing the on-chain contract addresses that represent the DAO's public system.

How to enter a chain: use CAIP-2 format. Ethereum mainnet is eip155:1, Polygon is eip155:137, and Polygon Amoy testnet is eip155:80002.

What the app does next: these contract references become services inside the DAO DID document. The MVP checks the address format but does not audit contract code.

At least one public DAO contract address is required. Use a CAIP-2 chainId plus an EVM address.

RSA 301-B evidence intake

What you are doing: supplying the public evidence a reviewer needs to understand the DAO's claimed compliance posture.

What the app does next: the server syntax-checks every URL, confirms attestations are true, stores the normalized checklist, and embeds it in the signed DAO DID document as an NHDAOComplianceChecklist service.

Important legal boundary: this MVP records submitted evidence. It does not certify that the DAO is legally compliant. A Secretary of State reviewer still approves, denies, or requests correction in the admin portal.

MVP evidence checklist for registry listing. The registry records these claims in the signed DAO DID document but does not certify statutory compliance.

Review and file

What happens when you click File registration: the browser sends the form to POST /api/file. If this Fly deployment has FILING_API_KEY set, the request must include the bearer token from the API key box above.

On the server: Express validates the filing, pins the governance bytes locally and to Arweave when configured, builds the DAO and registered-agent DID documents, signs both with the registry controller key, saves the record, and anchors document hashes on Polygon Amoy when configured.

After filing: you can inspect the public DID documents, run verification, and use the admin portal to approve the filing. Approval issues a new signed version of the DID documents.
